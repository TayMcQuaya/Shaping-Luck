# Building Alex — Phased Exploration Plan

Alex isn't a single product. It's a system of AI loops that compound on each other. Each phase adds a new loop, and each loop makes the previous ones smarter. The key insight: **we don't need to build the whole thing to start delivering value. Phase 0 can run on a spreadsheet and Claude.**

---

## Philosophy: Build the Smallest Loop That Delivers Magic

The temptation is to architect the whole system. Don't. Instead:

- Start with the loop that creates the most emotional impact with the least infrastructure
- Validate that people actually respond to AI-curated serendipity before building the engine
- Let real user data shape what the algorithm optimizes for — don't guess

---

## Phase 0 — The Manual Magic (Weeks 1-4)

_"Wizard of Oz it. The human IS the loop."_

### What We Build

Nothing. We run the system by hand using existing tools.

### How It Works

- **Onboarding**: Google Form with the 7 questions. Answers go to a spreadsheet.
- **Profile Interpretation**: Copy answers into Claude/ChatGPT with a prompt: _"Based on these 7 answers, generate a Luck DNA summary: core ambition, energy style, hidden interests, risk profile, aspirational identity, key tension, serendipity pattern."_
- **Meeting Pairings**: Before each meeting, paste all attendee Luck DNA summaries into an LLM prompt: _"Generate pairings for 3 rounds: Resonance (shared hidden dimensions), Stretch (useful difference), Catalyst (build a trio around one person's Q1 ambition). For each pair, write a one-sentence 'why you're meeting' and a custom conversation starter."_
- **Pre-Meeting Briefs**: Manually email each attendee their personalized brief (generated by the LLM).
- **Follow-Up**: After the meeting, prompt the LLM with meeting notes + attendee profiles to generate personalized follow-up messages.

### What We Learn

- Do people actually read the pre-meeting briefs?
- Do the pairings feel "magical" or forced?
- Which round (Resonance, Stretch, Catalyst) generates the most energy?
- What do people say in post-meeting feedback? What surprised them?
- How long does the manual process take? Where are the bottlenecks?

### Success Signal

People say some version of: _"How did you know to pair me with that person?"_

### Cost

$0 in infrastructure. ~3-5 hours of human time per meeting.

## Phase 1 — The First Real Loop (Months 1-2)

_"Automate the onboarding-to-profile pipeline. Keep pairings semi-manual."_

### What We Build

- **Onboarding flow** in the app (the 7 questions, nicely designed)
- **Luck DNA Generator** — an API endpoint that takes 7 answers and returns a structured profile
- **Profile database** — stores Luck DNA for all members (Supabase)
- **Embedding pipeline** — each profile gets vectorized for similarity/distance matching

### Architecture

```
User completes onboarding
        ↓
7 answers → LLM call (system prompt: Luck DNA schema)
        ↓
Structured JSON profile stored in Supabase
        ↓
Profile text → embedding model → vector stored in pgvector
```

### The LLM Prompt (Luck DNA Generator)

Takes the 7 raw answers and outputs:

```json
{
  "core_ambition": "...",
  "energy_style": "beeliner | observer | helper | open",
  "hidden_interests": ["...", "..."],
  "risk_profile": "high | moderate | cautious | deliberate",
  "aspirational_figures": ["..."],
  "aspirational_topics": ["..."],
  "key_tension": "...",
  "serendipity_style": "connector | explorer | lurker | builder",
  "enneagram_signals": {
    "likely_center": "heart | head | gut",
    "likely_stance": "dependent | aggressive | withdrawing",
    "noted_patterns": "..."
  },
  "quadrant_lean": "launchpad | forge | gilded_cage | bunker",
  "luck_dna_summary": "2-3 sentence human-readable summary"
}
```

### What's Still Manual

- Meeting pairings (but now the human has structured profiles to work with instead of raw text)
- Pre-meeting briefs (LLM-generated but human-reviewed before sending)
- Follow-up messages (LLM-drafted, human-approved)

### What We Learn

- Does the LLM consistently generate useful Luck DNA from the 7 answers?
- Do the embeddings cluster people in ways that feel right?
- What's the variance in profile quality? Do some answer combinations produce flat profiles?
- Which Luck DNA fields are actually useful for pairings vs. noise?

### Success Signal

Pairing quality goes up while human time per meeting goes down.

## Phase 2 — The Matching Engine (Months 2-4)

_"Automate the pairings. Humans review, not create."_

### What We Build

- **Matching algorithm** — takes a list of attendee IDs, outputs pairings for all 3 rounds
- **Brief generator** — takes a pairing and generates the personalized pre-meeting brief
- **Meeting dashboard** — facilitator sees all pairings, briefs, and prompts in one view

### Matching Logic

**Resonance Pairs** (Round 1):

```
For each attendee pair:
  - Cosine similarity on hidden_interests embeddings
  - Exact match bonus on aspirational_topics
  - Shared contradiction theme detection (LLM call)
  - Score = weighted sum → top pairs win
```

**Stretch Pairs** (Round 2):

```
For each attendee pair:
  - Cosine DISTANCE on full profile embeddings (want high distance)
  - BUT: run LLM check for "non-obvious thread" connecting them
  - Reject pairs where the LLM can't find a thread (pure randomness isn't stretch, it's chaos)
  - Score = distance × thread_strength → top pairs win
```

**Catalyst Trios** (Round 3):

```
For each attendee with a strong Q1 ambition:
  - Find people whose skills, industry, or network could unlock Q1
  - LLM call: "Given Person A's ambition [X], would Person B or C be useful? How?"
  - Build trios: one dreamer + two enablers
  - Rotate who's the "dreamer" across meetings so everyone gets a turn
```

**Luck Prompt Generator**:

```
Batch all attendee Q1 + Q6 answers
        ↓
LLM: "Find 2-3 thematic clusters in this group.
       Generate an opening question that touches the dominant theme
       without being so specific that it excludes anyone."
        ↓
Output: 1 Luck Prompt + rationale
```

### The Review Layer

All outputs go through a human facilitator who can:

- Swap a pairing that doesn't feel right
- Edit a brief that's off-tone
- Override the Luck Prompt
- Flag profiles that need re-onboarding (answers too thin)

### What We Learn

- How often does the human override the algorithm? On what basis?
- Do attendees notice a quality difference between manual and algorithmic pairings?
- Is the "non-obvious thread" detection reliable or does it hallucinate connections?
- How much does the Catalyst Round rotation matter for retention?

### Success Signal

Facilitator overrides < 15% of pairings. Post-meeting feedback scores stay the same or improve.

## Phase 3 — The Serendipity Crawler (Months 3-6)

_"Now Alex starts looking outward. The web becomes the opportunity space."_

### What We Build

- **Source registry** — curated list of crawl targets (event sites, job boards, grant pages, niche communities, podcast feeds, newsletters, subreddits)
- **Crawler agents** — scheduled scrapers that pull new content daily/weekly
- **Opportunity parser** — LLM that reads raw crawled content and extracts structured opportunity objects
- **Serendipity matcher** — compares opportunity embeddings against user Luck DNA embeddings
- **Luck Feed** — user-facing feed of 3-5 matched opportunities per week

### Architecture

```
Source Registry (curated URLs + RSS + APIs)
        ↓
Crawler Agents (scheduled, per-source)
        ↓
Raw content → Opportunity Parser (LLM)
        ↓
{
  "title": "...",
  "type": "event | job | grant | community | content | person",
  "domain": "biotech | design | finance | ...",
  "description": "...",
  "why_interesting": "...",
  "url": "...",
  "expires": "date or null"
}
        ↓
Embedding → pgvector
        ↓
Nightly job: for each user
  - Pull top 20 opportunities by cosine similarity to Luck DNA
  - Re-rank with LLM: "Score each opportunity 1-10 for serendipity
    (unexpected but meaningfully connected to this user's profile)"
  - Filter: keep only scores 7+
  - Generate one-line hook per opportunity: "Why this matters for YOU"
  - Push top 3-5 to Luck Feed
```

### The Serendipity Score

Not just relevance. The score is a tension between two forces:

- **Connection strength**: How clearly does this link to the user's profile?
- **Surprise factor**: How far outside their current orbit is it?

```
serendipity = connection_strength × surprise_factor
```

High connection + low surprise = "useful but boring" (LinkedIn already does this)
Low connection + high surprise = "random noise" (Twitter already does this)
High connection + high surprise = **serendipity** (this is what we're building)

### Source Strategy (Start Small)

Don't try to crawl the whole internet. Start with:

- 10-20 curated event aggregators relevant to the founding member base
- 5-10 niche newsletter RSS feeds
- Hacker News / specific subreddits for trend signals
- A few job boards (not for job matching — for "interesting company doing unexpected thing" signals)

Expand sources based on what users actually engage with in the Luck Feed.

### What We Learn

- What source types produce the highest serendipity scores?
- Do users engage with the Luck Feed? Click-through rates by opportunity type?
- Is weekly delivery the right cadence or do people want more/less?
- What's the false positive rate? (Opportunities surfaced that feel irrelevant)

### Success Signal

Users click on at least 1 Luck Feed item per week. Someone says _"How did it know I'd care about this?"_

## Phase 4 — The Follow-Up Loop (Months 4-6)

_"Alex doesn't forget. The system has memory and momentum."_

### What We Build

- **Connection graph** — tracks who met whom, in which round, with what outcome
- **Momentum tracker** — monitors whether users follow up on Catalyst Round commitments and Luck Feed items
- **Nudge engine** — generates and sends personalized follow-up messages at the right time
- **Re-calibration triggers** — detects when a user's profile might be stale and prompts a refresh

### The Nudge Engine

```
Triggers:
  - 24 hours post-meeting → send personalized recap + action items
  - 7 days post-meeting → check follow-up status, nudge if needed
  - User clicks Luck Feed item → 3 days later, "Did you explore [X]? Here's a related [Y]"
  - User's Q1 ambition keyword appears in news/events → instant nudge
  - 30 days since onboarding → "Anything changed? Quick re-calibration?"
  - Connection partner posts/does something relevant → cross-nudge

Nudge generation:
  Context (user profile + trigger event + connection history)
        ↓
  LLM: "Write a short, warm, specific nudge message.
        Tone: like a thoughtful friend who remembered something
        you mentioned. Not a bot. Not corporate."
        ↓
  Output → review queue or auto-send (based on confidence threshold)
```

### The Connection Graph

```
Nodes: Users
Edges: Meetings (with metadata)
  - round_type: resonance | stretch | catalyst
  - conversation_quality: self-reported (1-5)
  - followed_up: boolean
  - outcomes: [free text from follow-up check-ins]
  - shared_interests: [from pairing rationale]
```

This graph becomes the most valuable data asset in the system. Over time it tells us:

- Which pairing _types_ produce the most follow-ups?
- Are Resonance connections stickier than Stretch connections?
- Do Catalyst trios actually produce action?
- Who are the "super-connectors" in the network?

### What We Learn

- Do nudges increase follow-up rates vs. no nudges?
- What's the optimal nudge cadence before it feels annoying?
- Do people actually re-calibrate when prompted?
- Which edge types in the connection graph correlate with real-world outcomes?

### Success Signal

Follow-up rate after meetings exceeds 60% (vs. typical networking ~10-15%).

## Phase 5 — The Living System (Months 6-12)

_"Alex learns from its own history. The loops feed each other."_

### What We Build

- **Feedback integration** — post-meeting ratings, Luck Feed engagement, nudge response rates all flow back to improve matching and crawling
- **Profile evolution** — Luck DNA updates passively based on behavior, not just re-onboarding
- **Cross-member pattern detection** — Alex identifies emerging themes across the community and creates opportunities around them
- **Meeting format experimentation** — A/B test different round structures, prompt styles, group sizes based on outcome data

### The Flywheel

```
Better profiles → better pairings → better meetings
        ↓                                    ↓
  More engagement data              Higher retention
        ↓                                    ↓
  Smarter crawler targets           More members
        ↓                                    ↓
  Better Luck Feed                  Richer connection graph
        ↓                                    ↓
  More click/save/share data        Better pairing data
        ↓                                    ↓
        └──────── feeds back into ───────────┘
```

### What Makes This Defensible

- The connection graph is proprietary — no one else has this data
- The Luck DNA profiles get richer over time — hard to replicate
- The serendipity scoring model improves with every meeting and every Luck Feed interaction
- Network effects: the more members, the better the pairings, the better the experience

## Technical Stack (Suggested)

| Component             | Tool                                                           | Why                                       |
| --------------------- | -------------------------------------------------------------- | ----------------------------------------- |
| Database              | Supabase (Postgres + pgvector)                                 | Already in use, vector search built in    |
| LLM                   | Claude API (Sonnet for speed, Opus for deep analysis)          | Best at nuanced human interpretation      |
| Embeddings            | Voyage AI or OpenAI ada-002                                    | Good general-purpose embeddings           |
| Crawler               | Custom Python scripts + Firecrawl or Apify for complex sites   | Flexible, can start simple                |
| Scheduler             | Supabase Edge Functions + cron or simple Railway/Render worker | Lightweight, scales later                 |
| Nudge delivery        | Email (Resend or Postmark) + in-app notifications later        | Email first, app notifications in Phase 5 |
| Frontend              | Existing static site → evolve to Next.js/React when needed     | Don't over-engineer early                 |
| Facilitator dashboard | Simple admin page (HTML + Supabase queries)                    | Phase 2 deliverable                       |

## Budget Reality Check

| Phase   | Primary Cost                                     | Estimated Monthly |
| ------- | ------------------------------------------------ | ----------------- |
| Phase 0 | Human time only                                  | $0 (just sweat)   |
| Phase 1 | LLM API calls (onboarding)                       | $50-100           |
| Phase 2 | LLM API calls (matching + briefs)                | $200-400          |
| Phase 3 | Crawler infra + LLM calls (parsing + scoring)    | $500-800          |
| Phase 4 | LLM calls (nudges) + email delivery              | $300-500          |
| Phase 5 | All of the above + compute for pattern detection | $1,000-1,500      |

These are rough estimates for ~500 members. The LLM costs scale with member count and meeting frequency, but the per-unit cost drops as we cache more and prompt less.

## The One Rule

**Every phase must deliver a moment where someone feels seen by the system.** If it doesn't produce that feeling, it's infrastructure, not product. Infrastructure can wait. The feeling can't.
